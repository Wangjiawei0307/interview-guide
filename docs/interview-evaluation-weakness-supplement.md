# AI Agent 面试评估不足项补充回答

## 总体回应

如果面试官问：“评估里提到你这个项目还有一些不足，比如自动化测评、Agent 安全、向量数据库底层、MCP 底层机制、LLM 成本优化，你怎么看？”

可以这样回答：

> 我认可这些不足。这个项目当前更偏向一个可运行、可演示、业务闭环完整的 AI 面试平台，重点放在 RAG 知识库、Skill 可配置化、语音 Agent Loop、结构化输出和 Redis Stream 异步评估这些工程链路上。但如果要从课程项目或个人项目进一步走向生产级 Agent 系统，确实还需要在评测、安全、检索底层、协议化工具治理和成本控制上继续增强。我会把这些不足拆成五个方向来补齐。

---

## 1. 自动化质量评估体系不足

当前项目已经有一些基础能力：

- 出题和评估使用结构化输出，避免模型随意返回。
- RAG 无命中时返回固定提示，降低幻觉。
- 语音面试结束后会生成评估报告，并记录总分、逐题反馈、改进建议。
- 我在调试时会人工构造样例，观察 Query Rewrite、召回片段、最终回答是否符合预期。

但不足是：

> 当前主要还是人工样例测试，没有形成一套可重复执行、可量化对比的 Eval 测评体系。

面试版补充可以这样说：

> 如果继续优化，我会给这个项目补一套离线 Eval 测评集。比如针对 RAG 模块，构造三类样本：能直接命中的问题、需要 Query Rewrite 的多轮追问、知识库无答案的问题。每条样本都保存标准问题、期望命中文档片段、期望答案要点，然后定期跑检索命中率、TopK 召回质量、无命中识别准确率、答案忠实度和结构化输出通过率。  
>
> 对面试出题模块，我会评估题目是否符合 skill 分类权重、难度是否匹配、题目是否重复、是否结合简历或 JD。对语音面试模块，我会额外统计 ASR 首字延迟、LLM 首 token 延迟、TTS 合成耗时、整轮 turn latency。这样就能把“感觉回答不错”变成一组可追踪的指标，也方便后续更换模型或调整 Prompt 时做回归测试。

可以强调一句：

> 这个方向本质上是把 Agent 应用从“能跑”升级到“可评估、可回归、可持续优化”。

---

## 2. Agent 安全能力不足

当前项目已经有部分安全边界：

- Prompt 中区分了系统角色、用户输入、简历、JD、知识库内容。
- `PromptSanitizer` 会对用户输入做边界包装，降低 Prompt 注入影响。
- 工具调用主要用于加载 Skill，属于只读工具，不让模型直接改数据库或执行高风险操作。
- RAG 无命中不会让模型自由发挥，而是返回固定提示。

但不足是：

> 当前还没有形成完整的 Agent 安全体系，比如 Prompt Injection 检测、工具权限控制、敏感操作审批、越权访问防护等。

面试版补充可以这样说：

> 如果继续加强 Agent 安全，我会从三个层面做。第一是输入侧，把用户问题、简历、JD、知识库内容全部当成不可信输入，做 Prompt Injection 检测和边界隔离，明确告诉模型这些内容不能覆盖系统指令。第二是工具侧，给每个工具定义权限等级，比如只读工具、写操作工具、外部调用工具，高风险工具必须经过后端白名单和参数校验，不能完全交给模型自由调用。第三是数据侧，RAG 检索时必须带 userId、kbId 等租户隔离条件，防止一个用户查到另一个用户的知识库。  
>
> 在这个项目里，Skill 工具目前是只读的，风险相对可控；但如果未来接入 MCP、文件系统、日程、邮件或数据库写操作，就必须引入工具权限、审计日志和人工确认机制。

可以强调一句：

> Agent 安全的核心不是“相信模型不会乱来”，而是后端要把权限边界、数据边界和副作用边界控制住。

## 3. 向量数据库底层原理需要补强

当前项目使用的是：

- PostgreSQL + pgvector
- 1024 维向量
- Cosine 相似度检索
- 文档解析、切分、Embedding、入库、TopK 检索、RAG 问答

但不足是：

> 当前更多停留在应用层，会用 pgvector 做相似度检索，但对 HNSW、IVFFlat、混合检索、Rerank 等生产级检索优化理解还不够深入。

面试版补充可以这样说：

> 这个项目当前用 pgvector 主要解决“知识库文档能被语义召回”的问题，适合数据量不大、架构简单的阶段。如果数据规模上来，我会进一步优化索引和召回策略。比如 pgvector 里可以根据数据规模选择 HNSW 或 IVFFlat，HNSW 查询速度和召回效果通常更好，但索引构建和内存成本更高；IVFFlat 更依赖 list 数和 probes 参数，需要结合数据量调参。  
>
> 另外，纯向量检索对关键词、专有名词、代码类问题不一定稳定，所以我会考虑做混合检索：向量召回负责语义相似，BM25 或全文索引负责关键词匹配，再用 RRF 做结果融合。如果对回答质量要求更高，还可以在 TopK 之后加 Reranker，对候选片段重新排序，最后再进入 Prompt。  
>
> 在这个项目里，短期可以先把 Query Rewrite、TopK、minScore、文档分块大小和重叠窗口调好；长期再引入混合检索和 Rerank。

可以强调一句：

> RAG 的核心瓶颈很多时候不是模型，而是“是否召回了正确证据”。

## 4. MCP 协议底层机制理解需要深入

当前项目的真实情况：

- 没有真正接入 MCP。
- 工具调用主要是 Spring AI Tool Calling。
- Skill 作为受控只读工具，服务于语音面试角色和提问风格加载。

不足是：

> 当前对 MCP 的回答更多停留在工程价值层，比如工具标准化、解耦、复用，但对协议底层和生产部署细节还可以更深入。

面试版补充可以这样说：

> 这个项目当前没有接入 MCP，主要是因为它还是单体 Spring Boot 应用，RAG 检索和 Skill 加载都在本地后端完成，本地方法调用的延迟和可控性更好。  
>
> 如果未来引入 MCP，我会把它理解成 Agent 与外部工具之间的标准化协议层。底层上，MCP 基于 JSON-RPC 思路组织请求和响应，工具通过 schema 描述输入输出，客户端可以发现工具、调用工具、拿到结构化结果。传输层可以是本地 stdio，也可以是远程 HTTP 类方式。  
>
> 但引入 MCP 也会带来成本：链路变长、延迟增加、MCP Server 成为新的故障点，还要做工具权限、版本管理、审计和多租户隔离。所以我不会为了追新技术强行引入 MCP。只有当 RAG 检索、日程、文件、代码仓库等工具能力需要被多个 Agent 或多个系统复用时，我才会把它们封装成 MCP Server。

可以强调一句：

> MCP 的价值在于工具标准化和跨系统复用，不是所有单体项目都必须上 MCP。

---

## 5. LLM 成本优化策略不足

当前项目已经做了一些间接控制：

- RAG 无命中时不调用模型自由发挥，减少无效生成。
- 出题、评估等场景使用结构化输出，失败后有限重试。
- 语音链路使用流式输出，提升交互体验。
- Redis Stream 异步评估避免用户一直阻塞等待。

但不足是：

> 当前还没有系统性做 LLM 成本治理，比如模型路由、Token 预算、Prompt 缓存、语义缓存、长上下文压缩等。

面试版补充可以这样说：

> 如果要把这个项目做成生产级，我会从四个方面优化 LLM 成本。第一是模型路由，不同任务用不同模型，比如 Query Rewrite、分类、简单结构化解析可以用小模型，最终评估报告和复杂面试追问再用大模型。第二是 Token 预算管理，RAG 召回片段不能无限塞进 Prompt，要按相关性排序、去重、裁剪，并控制历史对话窗口。第三是缓存，对知识库问答可以做语义缓存，对固定 Skill Prompt、系统 Prompt 可以做 Prompt Cache 或模板缓存。第四是失败重试控制，结构化输出重试要限制次数，并区分格式错误、超时、供应商限流、余额不足等错误类型，避免盲目重试烧成本。  
>
> 在这个项目里，比较容易落地的是给 `LlmProviderRegistry` 增加按场景路由，比如 `rewrite`、`question-generate`、`evaluation`、`voice-dialogue` 分别配置模型；再给 RAG 模块加 Token 预算裁剪和召回片段去重。

可以强调一句：

> LLM 成本优化不是单纯换便宜模型，而是“任务分级、上下文压缩、缓存复用、有限重试”的组合。

---

## 面试中可以统一总结

最后可以用这段话收束：

> 所以我认为这个项目当前已经完成了 AI Agent 应用从 0 到 1 的核心闭环，包括 RAG、Skill、Tool Calling、语音流式链路、结构化输出和异步评估。但如果从工程成熟度看，还需要继续补齐五类能力：第一是 Eval 自动化，把效果变成可量化指标；第二是 Agent 安全，把模型权限和数据边界控制住；第三是检索底层优化，从单纯向量检索升级到混合检索和 Rerank；第四是 MCP 协议化工具治理，在多 Agent、多系统复用场景下再引入；第五是 LLM 成本治理，用模型路由、Token 预算和缓存降低长期运行成本。这样项目就能从一个功能完整的 AI 面试平台，进一步演进成更接近生产级的 Agent 工程系统。
